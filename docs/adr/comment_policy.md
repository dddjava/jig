# コメントには背景理由を書き、経緯は書かない

## 状況 (Context)

コードやテストのコメントに「このコードがなぜ現在の形なのか」を伝える情報を残す際、背景理由と経緯が混在しがちです。issueレビュー起点の修正では、コメントに `#1152` のようなissue番号が経緯の言及として書き込まれることがありました。

- **背景理由**: 現在のコードを正当化する、コードだけでは読み取れない制約や判断。コードが存在する限り有効。
- **経緯**: どのissueで見つかったか、どう変遷してきたか。修正が完了した時点で、現在のコードを理解するためには不要になる情報。

経緯はgit履歴・コミットメッセージ・issueが既に記録しており、コメントに重複して残すと、読み手が「この参照を辿る必要があるのか」を判断するノイズになります。

## 決定 (Decision)

1. **背景理由はコメントに書く**
   - 現在のコードに至る理由（制約・トレードオフ・採用しなかった代替案の却下理由など）は、経緯の有無に関係なくコメントに記述してよく、コードだけで読み取れない場合は記述すべきです。
   - 理由はコメント本文だけで完結させます。issueやPRを読まないと理解できない書き方をしません。

2. **経緯はコメントに書かない**
   - 「issue #NNNN で見つかった」「以前は〜だった」のような経緯はコミットメッセージ・トレーラー（`Closes: #N` 等）に記述し、コメントには残しません。
   - 恒久的な対応が完了したコードにissue番号への参照を残しません。

3. **一時的なもの・暫定回避には issue 参照を書く**
   - 暫定回避（workaround）や、解消待ちの既知の問題を抱えたコードなど、経緯自体に価値がある場合は、追跡先としてissue番号をコメントに記述すべきです。この参照は恒久対応の完了とともに削除します。

## 根拠・背景 (Rationale)

- **コメントの自己完結性**
  issue参照が必要になるのは、理由の説明が本文で完結していないサインです。理由を書き切れば参照は不要になります。

- **記録の役割分担**
  経緯はコミットメッセージ・git履歴・issueが担う領域です。コメントに重複させると、コードの変更とともに陳腐化し、メンテナンスされないまま残ります。

- **参照のシグナル性維持**
  issue参照を「未解決の何かがある」印に限定することで、参照を見つけた読み手は「対応すべきものが残っている」と判断できます。恒久対応済みのコードにも参照が散在すると、このシグナルが薄まります。

## トレードオフ (Trade-Offs)

- **経緯を知りたい場合はgitを辿る必要がある**
  コメントからissueに直接飛べなくなりますが、`git log` / `git blame` からコミットメッセージ経由で到達できます。

- **一時的か恒久的かの判断が必要**
  コメントを書く時点で、その対応が暫定回避なのか恒久対応なのかを意識する必要があります。この判断自体が設計の明確化に寄与します。

## 結論 (Consequences)

コメントは「現在のコードを理解するための背景理由」に純化され、issue参照は「解消待ちの何かがある」ことを示すシグナルとして機能します。経緯を知りたい場合はgit履歴を辿ります。
