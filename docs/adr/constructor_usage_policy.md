# コンストラクタの使い分け方針

## 状況 (Context)
オブジェクト生成時に、コンストラクタ内で値の変換や分岐などのロジックを行う実装はしばしば見られます。
しかし、コンストラクタが初期化以上の責務を持つと、意図の把握やテストの焦点が曖昧になり、生成ルールの変更時にも影響範囲が広がりやすくなります。

このため、コンストラクタの責務と変換ロジックの配置を明確にします。

## 決定 (Decision)
1. **コンストラクタは基本的にロジックを書かない**
   - コンストラクタは引数を受け取り、インスタンスフィールドに保持する責務に限定します。

2. **引数の変換ロジックはファクトリメソッドに置く**
   - 生成時に必要な変換（整形、パース、派生値の計算、分岐など）は、コンストラクタではなくファクトリメソッドで行います。

## トレードオフ (Trade-Offs)
以下のトレードオフを考慮します：

1. **メリット**
   - コンストラクタの責務が明確になり、可読性が向上する
   - 変換ロジックのテスト対象をファクトリメソッドに集約できる
   - 生成ルール変更時の影響箇所を限定しやすい

2. **デメリット**
   - ファクトリメソッドの追加により、APIの入口が増える
   - 単純な生成でも、呼び出し側でファクトリ利用を意識する必要がある

## 採用ガイドライン (Guidelines)
以下のルールに従います：

1. **コンストラクタの責務を初期化に限定する**
   - 引数チェックや変換、派生値計算などのロジックは原則として記述しません。
   - 受け取った引数は、そのまま対応するインスタンスフィールドに保持します。

2. **変換が必要な場合はファクトリメソッドを使う**
   - 変換後の値をコンストラクタに渡す公開ファクトリメソッド（例: `of`, `from`）を用意します。
   - ファクトリメソッドは「生成ルールの窓口」として扱います。

3. **呼び出し側の意図を明確にする**
   - 変換を伴う生成はファクトリメソッド経由に統一し、コンストラクタ直接呼び出しと混在させないようにします。

## 実装例
### 避ける例（コンストラクタで変換）
```java
public final class UserName {
    private final String value;

    public UserName(String rawValue) {
        this.value = rawValue == null ? "" : rawValue.trim().toLowerCase();
    }

    public String value() {
        return value;
    }
}
```

### 推奨例（ファクトリメソッドで変換）
```java
public final class UserName {
    private final String value;

    private UserName(String value) {
        this.value = value;
    }

    public static UserName from(String rawValue) {
        String converted = rawValue == null ? "" : rawValue.trim().toLowerCase();
        return new UserName(converted);
    }

    public String value() {
        return value;
    }
}
```

## 結論 (Consequences)
- コンストラクタが「受け取って保持する」責務に集中し、設計の一貫性が高まります。
- 変換ロジックをファクトリメソッドに分離することで、生成ルールの見通しと保守性が向上します。
- オブジェクト生成の入口が明確になるため、将来的な仕様変更にも追従しやすくなります。
