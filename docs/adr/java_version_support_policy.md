# Javaバージョンサポートポリシー

## 状況 (Context)

JIG には性質の異なる二つの Java バージョン要件があります。

- **JIG 自体を動かす環境**: 実行には Java 21 以降が必要で、ビルドの toolchain は 21 に固定しています。
- **JIG が読み取る対象**: README で「Java8のプロダクトであっても、Java21で実行するJIGで解析することは可能です」と約束しています。JIG はバイトコードを解析するため、解析対象の Java バージョンは JIG の実行環境とは独立です。

この二つは混同されやすく、実行環境の要件だけがドキュメント化されている一方、解析対象の下限がどこまでかは明文化されていませんでした。さらに、テストの fixture は `jig-core` のテストコンパイル成果物として得ているため toolchain 21 でしか作られておらず、**Java 8 のクラスファイルを解析できるという約束は検証されていません**。

解析対象の下限を検証するには、その下限でコンパイルしたクラスファイルが必要です。`--release 8` は JDK 25 でも利用できますが、廃止予定である旨の警告が出ます。

```
警告: [options] ソース値8は廃止されていて、今後のリリースで削除される予定です
```

いずれ指定できなくなったとき、サポートを続けるか終了するかの判断基準が必要です。

## 決定 (Decision)

1. **JIG 自体の実行環境は Java 21 以降**とし、ビルドの toolchain は 21 に固定します。実行環境の下限を上げるのは、依存する Gradle・Spring Boot・ライブラリの要求に基づく別の判断とします。

2. **解析対象は Java 8 以上のクラスファイル**とします。JVM 言語で生成されたクラスファイルも同様に扱います。

3. **解析対象バージョンの検証点を三つに固定**します。同一の fixture ソースを `--release` で切り替えてコンパイルし、同じ解析結果が得られることを検証します。
   - サポートする下限（現在は 8）
   - ビルドの toolchain と同じバージョン（現在は 21）
   - 最新の LTS（現在は 25）

4. **下限を切る条件を、ツールチェーンで再現できるかどうかに一致させます**。最新 LTS の javac で `--release <下限>` が指定できなくなった時点で、そのバージョンの解析対象サポートを終了し、下限を次のバージョンへ繰り上げます。

5. **事前コンパイル済みのクラスファイルを fixture に持ってサポートを延命しません**。検証できる範囲とサポートを約束する範囲を一致させ、検証できない約束を残さないためです。

## トレードオフ (Trade-Offs)

1. **メリット**
   - README の約束が自動検証の対象になります
   - サポート終了の判断が、コンパイル時の警告という観測可能な事実に紐付き、議論を要しません
   - 検証点が三つに固定されるため、新しい Java がリリースされても検証コストが増えません
   - fixture が常にソースから再現できるため、内容の確認と変更が容易です

2. **デメリット**
   - 最新 LTS のクラスファイルを作るため、ビルドに toolchain の追加取得が必要になります
   - `--release 8` の廃止に伴い、JIG 側の都合と無関係にサポートを切ることになります
   - 三バージョン分のコンパイルによりビルド時間が増えます
   - 下限より古いクラスファイルを解析できる場合でも、サポートとしては約束しません

## 採用ガイドライン (Guidelines)

1. **検証点の維持**
   - 新しい LTS がリリースされたら、検証点の「最新の LTS」を更新します。それ以外の検証点は増やしません。
   - toolchain を上げたときは、検証点の「ビルドの toolchain と同じバージョン」を合わせます。

2. **サポート終了の手順**
   - `--release <下限>` の指定が最新 LTS の javac でエラーになった時点で終了を判断します。警告が出ている間は終了しません。
   - 終了時は fixture の対象バージョンを新しい下限へ入れ替え、README と本 ADR を更新します。
   - 終了はリリースノートで明示します。

3. **バージョン依存の記述**
   - 実行環境の要件と解析対象の要件を、ドキュメント・エラーメッセージ・設定項目で明確に区別します。「Java 21 が必要」とだけ書かず、どちらの要件かを示します。

## 実装例

fixture のコンパイルは、同一ソースに対して `--release` を切り替えたタスクを登録して行います。toolchain は最も新しい検証点に合わせ、そこから各バージョンのクラスファイルを生成します。

```kotlin
// jig-test-fixtures の代表プロジェクトのうち、解析対象バージョンを検証するもの
listOf(8, 21, 25).forEach { release ->
    tasks.register<JavaCompile>("compileBytecodeCompat$release") {
        source = fileTree("projects/bytecode-compat/src/main/java")
        destinationDirectory = layout.buildDirectory.dir("fixtures/bytecode-compat/classes-$release")
        options.release = release
        classpath = files()
    }
}
```

## 結論 (Consequences)

- README の約束と検証内容が一致し、解析対象バージョンの退行を PR で検知できます
- サポート終了の時期が JDK 側の都合で決まるため、JIG 側で維持コストを抱え込みません
- Java 8 サポートは `--release 8` が使える限り継続し、使えなくなった時点で下限が繰り上がります
- ビルドに toolchain の追加取得が必要になり、CI が JDK 21 単一ではなくなります
