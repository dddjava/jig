# テストは利用者が観測する契約で組み立てる

## 状況 (Context)

JIG のテストは実装クラス単位で積み上がってきました。`jig-core` の `src/test/java` にある約 225 の Java ファイルのうち、テストクラスは約 50 で、残りの約 170 は解析対象として使う fixture です。この構成には次の課題があります。

- **テストの単位が利用者の関心と対応していない**。利用者が信頼するのは内部オブジェクトの生成手順ではなく、入力したプロジェクトから正しいドキュメントを再現可能に出力できることです。実装クラス単位のテストは、それが壊れたときに利用者へ何が起きるかを示しません。
- **fixture がテストのビルド成果物に癒着している**。`testing.TestSupport` は `DefaultPackageClass.class` の位置からテスト出力ディレクトリを逆算し、`build/classes/java/test/stub` と `src/test/java/stub` の対を組み立てています。fixture の所在がビルドレイアウトの暗黙知になっており、`jig-cli` や `jig-gradle-plugin` から同じ入力を再利用できません。
- **検証できていない公開契約がある**。README は Java 8 以上のプロダクトを解析できると約束していますが、fixture は toolchain 21 でしかコンパイルされていません。
- **調査コードとテストが同居している**。`learning` 配下の調査コードが CI で品質保証として実行されています。

一方で、現在の保証水準には下げてはならないものがあります。Gradle プラグインの TestKit テストは `SupportGradleVersion` の全バージョンを PR 時点から実行しており、これは `docs/adr/gradle_version_support_policy.md` の決定に基づくものです。

## 決定 (Decision)

テストを、実行技術ではなく**利用者が観測する契約**と**隔離の度合い**で組み立てます。

### 1. すべてのテストを四つの契約のいずれかに結び付ける

どれにも結び付けられないテストは作りません。

| 契約 | 失敗時に利用者へ起きること | 主な検証手段 |
| --- | --- | --- |
| 解析契約 | Java・クラス・設定ファイルからモデルを誤認識する | 単体、統合、代表プロジェクトのゴールデンテスト |
| 出力契約 | HTML・データ JS・JSON・アセットが壊れる、または内容が変わる | スナップショット、構造検証、ブラウザ E2E |
| 起動契約 | CLI または Gradle タスクから実行できない | ブラックボックス E2E、Gradle TestKit |
| 互換性契約 | サポートする JDK・OS・Gradle・ブラウザで動かない。サポートする Java バージョンでコンパイルされたクラスファイルを解析できない | 互換性マトリクス、複数バージョンでコンパイルした fixture、リリース候補検証 |

互換性契約には二つの軸があります。JIG 自体を動かす環境（JDK・OS・Gradle・ブラウザ）と、JIG が読み取る対象（クラスファイルの Java バージョン）です。サポート範囲は軸ごとの ADR に従います。

| 軸 | ADR |
| --- | --- |
| 実行 JDK・解析対象の Java バージョン | `java_version_support_policy.md` |
| Gradle | `gradle_version_support_policy.md` |
| ブラウザ | `browser_support_policy.md` |

### 2. テストを五つの階層に分類する

同じ振る舞いを複数階層で重複させません。下位層は原因を特定し、上位層は結線と利用者体験を保証します。

| 階層 | 対象 | 許可する依存・I/O | 代表的な検証 | 実行頻度 |
| --- | --- | --- | --- | --- |
| Unit | 値オブジェクト、判定、変換、ドメイン規則、画面資産の純粋関数 | 実ファイル、ネットワーク、Gradle、ブラウザを使わない | 境界値、同値類、不変条件、例外契約 | 全 PR |
| Component | 一つのアダプターまたはアプリケーション・ユースケース | テンポラリファイル、埋め込みパーサー、テスト用実装まで | 読み込みからモデル化、モデルから一種類の出力まで | 全 PR |
| Contract | JIG が生成・消費する公開形式 | バージョン管理された fixture のみ | JSON/データ JS/HTML のスキーマ・必須要素・互換性、同一入力に対する同一出力 | 全 PR |
| E2E | CLI、Gradle プラグイン、生成済みサイト | 実プロセスと隔離した代表プロジェクト | コマンド実行、成果物、主要画面操作 | 全 PR |
| Compatibility | サポート対象の組合せ | 実 JDK/OS/Gradle/ブラウザ、複数バージョンでコンパイルした fixture | 同一の公開シナリオ、解析対象クラスファイルのバージョン差 | Gradle と解析対象バージョンは PR、OS は main、その他は release |

決定性（同一入力から同一出力が得られること）に独立した階層は設けず、Contract の検証項目として扱います。性能とメモリ消費には自動の合否判定を置かず、必要時に計測します。

解析対象クラスファイルのバージョン差は、実行環境の組合せと違い CI のマトリクスを増やさずに検証できます。fixture のコンパイル時に `--release` を切り替えるだけで済むため、Compatibility に属しながら PR で実行します。

### 3. fixture を独立モジュール `jig-test-fixtures` に集約する

Gradle の `java-test-fixtures` 機能ではなく独立モジュールとし、公開対象から外します。fixture は三モジュールから参照されるため一箇所に集める必要があり、公開する `jig-core` のライフサイクルと依存関係から完全に分離するためです。他モジュールが依存できるのは fixture の展開・正規化 API とリソースだけで、本番コードへの依存は禁止します。

### 4. 代表プロジェクトを専用のソースセットでコンパイルする

JIG の入力はクラスファイルであり、Java ソースをリソースとして置くだけでは解析対象になりません。fixture API がクラス出力ディレクトリとソースディレクトリの対を返し、利用側はパスを組み立てません。配置と追加の手順は `jig-test-fixtures/README.md` にあります。

### 5. 重い検証を `build` に載せない

`check` は Unit/Component/Contract まで、`qualityCheck` が `check`・`e2eTest`・`jacocoTestReport` を束ねます。`check` を `qualityCheck` に依存させると `build` に E2E が載るため、依存の向きを逆にしません。

### 6. 生成物の検証は構造・意味を優先する

全ファイルの無差別な文字列比較をやめ、JSON は構造比較、HTML は DOM の必須構造・リンク・アクセシビリティ属性で比較します。同一入力から同一出力が得られること（決定性）は、二度実行して突き合わせる形で確認し、実行時刻に由来する値は正規化し、計測値そのもの（メトリクス）は対象から外します。

スナップショットは、構造では表せないが固定したい図や大きな文章に限って使います。更新は専用タスクで明示的に行い、通常のテスト実行中には更新しません。

### 7. 調査コードを実行対象から外す

`learning` は CI で品質保証として実行しません。将来の仕様を固定したくなった時点で、目的に合う階層へ昇格します。

## トレードオフ (Trade-Offs)

1. **メリット**
   - テストが壊れたときに、利用者へ何が起きるかが失敗メッセージから追えます
   - fixture が三モジュールの共通資産になり、CLI・Gradle プラグイン・ブラウザ検証で同じ入力を使えます
   - 解析対象・出力形式・対応環境を追加するとき、どこにどの粒度のテストを置くかが一意に決まります
   - 重い検証が `build` から分離され、開発中のフィードバックが速くなります

2. **デメリット**
   - モジュールとソースセットが増え、ビルド構成が複雑になります
   - fixture のコンパイル定義を自前で持つ必要があります
   - 既存テストの分類と移動に継続的な作業が必要です
   - 契約への結び付けを求めるため、テストを書く前の判断が一手増えます

## 採用ガイドライン (Guidelines)

1. **テストを追加するとき**
   - 「これが壊れると利用者に何が起きるか」を先に言葉にします。言えない場合はテストを書きません。
   - 答えた内容から階層を選びます。原因の特定が目的なら下位層、結線と利用者体験の保証が目的なら上位層です。
   - 同じ振る舞いが下位層で既に押さえられている場合、上位層で繰り返しません。

2. **入力を用意するとき**
   - 代表プロジェクトを `jig-test-fixtures` に置き、一つのプロジェクトに一つの意図を持たせます。
   - 既存の代表プロジェクトの組み合わせや、テスト内での加工で表せる入力には新しい fixture を作りません。マルチモジュール構成は既存プロジェクトを並べて展開すれば足り、壊れたクラスファイルはテスト内で書き出すほうが、リポジトリに壊れたバイナリを置くより扱いやすいためです。
   - 実ライブラリを読み取る契約は、ライブラリの最小公開 API を模した fixture を優先します。実ライブラリとの結線確認だけを Compatibility または E2E に一件置きます。
   - 期待値を固定する検証の入力に、JIG の自己解析結果を使いません。入力が製品コードと共に動くため、無関係な変更で期待値が壊れます。自己解析は大きな入力に対する挙動の確認にだけ使います。

3. **観測点を作るとき**
   - テストの都合で本番コードの可視性を上げません。必要な観測点は公開契約か、専用に設計された境界に置きます。
   - 時刻・絶対パス・ランダム値・OS の区切り文字を成果物に残さず、避けられない値は比較前に正規化します。

4. **自動検査を追加するとき**
   - 本番コードのパッケージ間依存は自動検査しません。規則を機械的に定義する便益より、規則と実装の乖離を保守する手間が上回るためです。
   - カバレッジは JaCoCo で収集しますが、しきい値による合否判定を置きません。変更行カバレッジ・公開シナリオの網羅・失敗の再現性・テスト時間を継続監視します。
   - 性能・変異解析のような重い自動検証は導入しません。パイプラインを増やして検証を分散させることもしません。運用コストに見合わないためです。

5. **モジュールごとの置き場所**

   | 対象 | 置く階層 | 置かないもの |
   | --- | --- | --- |
   | `jig-core` のドメイン規則・識別子・関係導出・フィルタリング・設定解釈 | Unit | private 実装の構造 |
   | ASM・JavaParser・MyBatis・Spring Data・Git・ファイルシステムのアダプター | Component（成功・空入力・不正入力・境界入力を各一つ以上） | — |
   | ドキュメント生成 | Contract（入力モデル → 公開成果物） | HTML・データ JS・JSON の生成順序、内部 DTO |
   | `JigExecutor` を通す代表シナリオ | E2E（少数） | 各アダプターの全分岐 |
   | CLI の引数解釈・デフォルト・エラー表示 | Unit/Component | — |
   | CLI の起動 | E2E（別プロセスで標準出力・終了コード・生成物を確認） | Spring の内部 Bean 構成、private メソッド |
   | Gradle 拡張 DSL からタスク設定への変換 | Unit | — |
   | Gradle TestKit シナリオ | E2E（`src/test` に置いたまま） | — |
   | JavaScript の表示規則・データ変換・イベント配線 | Unit（DOM を最小化） | — |
   | 生成 HTML とデータ JS の必須 data 属性・ID・JSON 形式・エスケープ規則 | Contract | — |

   TestKit テストを `src/functionalTest` へ分離しないのは、分離しても実行タイミングが変わらず（PR で全サポートバージョンを実行する）、ソースセットを増やす便益がないためです。

6. **依存してはいけないもの**
   - 実ネットワーク、現在時刻、共有ディレクトリ、テスト実行順序に依存しません。
   - Unit はネットワークと実プロセスを起動しません。これを許可されるのは E2E だけです。
   - `jig-test-fixtures` を本番成果物の依存グラフに入れません。Gradle の依存宣言で構造的に担保し、`jig-core` の publication に含めません。
   - テスト名の主語に実装メソッド名や issue 番号を置きません。入力・操作・観測可能な結果を表します。

## 実装例

階層の選択は「壊れたときに利用者へ起きること」から導きます。

| 検証したいこと | 契約 | 階層 |
| --- | --- | --- |
| パッケージ名の親子判定 | 解析契約 | Unit |
| ASM がジェネリクスのシグネチャを読める | 解析契約 | Component |
| 生成した JSON に必須キーが揃っている | 出力契約 | Contract |
| Java 8 でコンパイルしたクラスを解析できる | 互換性契約 | Compatibility |
| `jig-cli.jar` を実行するとサイトが出力される | 起動契約 | E2E |

fixture の入力は API から受け取り、パスを組み立てません。

```java
// NG: ビルドレイアウトを前提にパスを逆算する
var classes = Paths.get(TestSupport.defaultPackageClassURI()).resolve("stub");

// OK: fixture モジュールの API から受け取る
var project = JigFixtures.project("bytecode-compat");
var sourceBasePaths = new SourceBasePaths(
        new SourceBasePath(List.of(project.classes(21))),
        new SourceBasePath(List.of(project.sources())));
```

テストの入口は隔離度と所要時間が名前から分かるようにします。実行方法は CLAUDE.md の「テスト実行ポリシー」を参照してください。

| 入口 | 内容 |
| --- | --- |
| `./gradlew test` | 全モジュールの Unit/Component と Gradle TestKit |
| `./gradlew contractTest` | 公開形式の Contract |
| `./gradlew e2eTest` | CLI と最小の生成シナリオ |
| `./gradlew qualityCheck` | `check`・E2E・カバレッジレポート。PR CI が呼ぶ |
| `npm run test` | Web の Unit/Component |
| `npm run test:contract` | Java の Contract と、その生成サイトを読む Web の Contract |
| `npm run test:full` | 開発中の標準入口 |
| `npm run test:pr` | CSS lint と Web 全テスト、`clean build qualityCheck` |

## 結論 (Consequences)

- テストの失敗が、利用者に起きる不具合として読めるようになります
- fixture が共通資産になり、モジュールをまたいだ入力の再利用ができます
- 未検証だった公開契約（解析対象の Java バージョンなど）が検証対象に入ります
- ビルド構成は複雑になりますが、重い検証が明示タスクへ分離されるため開発中の実行時間は延びません
- 既存テストの移行は段階的に行い、各段階で「新しい層に同等以上の公開契約テストがあり、CI で安定していること」を完了条件とします
